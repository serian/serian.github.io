hugoの管理

Jul 7, 2016   #misc  # 

Github Pagesとhugoについていくつかのサイトを読むと、どうもリポジトリを二つ作る流儀があるらしい。
というかそっちの方が多いような。
そして”gh-pages”とかいうブランチを作ったりという方式も。

gh-pagesを作るかどうか

User, Organization, and Project Pages - User Documentationを読むと、これはどうやらプロジェクト毎のGithub Pagesを作る時の形式らしい。
で、Github Pagesとしては、そのプロジェクト名が、
> username.github.io/<プロジェクト名> みたいな感じでディレクトリになるみたいな気がする。

今回みたいに全く独立のサイト、User & Organization Pages を作って、ルート直下にファイルが場合は別によさそう。

リポジトリを二つ作って管理するか

要は、元のmarkdownファイルや*Hugoのシステム関係のファイル*とHugoで作った*webサイトとして公開するファイル*、つまりpublicディレクトリを別に管理するってことのようだ。
だから、前者をbitbuckets、後者をgithubで、って人もいた。

Hugo本家のチュートリアルHugo - Hosting on GitHub Pagesにも、インプットとアウトプットを分けるのはナイスな管理とか書いてある。

git subtreeを使う流儀

gitで外部moduleを扱う方法(subtree) - Qiitaを読むと、単純に考えて、ファイルシステムで言うシンボリックリンクのような仕組みっぽい。

本家のチュートリアルのようなproject pagesの場合、masterとgithub pages用のgit-pagesという二つのブランチがあって、そこではいちいちブランチ移動して更新するのは面倒だとしてある。
そこで、git-pagesのrootを、masterにsubtreeしてミラーを作って、masterブランチで更新していく対処法が書いてあるような気がするが、gitもgithubも単純なことしか知らないのでよくわからない。
